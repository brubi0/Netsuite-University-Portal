Script Anatomy

Understanding the complete structure of a SuiteScript file

üìã The Comment Block

Every SuiteScript file begins with a comment block containing special JSDoc-style tags that NetSuite reads to understand the script:

/**
 * @NApiVersion 2.1
 * @NModuleScope SameAccount
 * @NScriptType UserEventScript
 * 
 * @see https://docs.oracle.com/en/cloud/saas/netsuite/ns-online-help/...
 * @author Your Name
 */

Required Tags

Tag Values Purpose
@NApiVersion 2.1, 2.0, 2.x, 1.0 Specifies SuiteScript version. Use 2.1 for modern JavaScript features.
@NScriptType UserEventScript, ClientScript, Suitelet, etc. Tells NetSuite what type of script this is.
@NModuleScope SameAccount, Public Controls which scripts can import this as a dependency.

üí° SuiteScript 2.1 Benefits

Version 2.1 enables modern JavaScript features: arrow functions, template literals, destructuring, const/let, and async/await. Always use 2.1 for new development.

üì¶ The Define Block

SuiteScript uses the AMD (Asynchronous Module Definition) pattern. The define() function loads module dependencies:

define(['N/record', 'N/search', 'N/log'],
    /**
     * @param {record} record
     * @param {search} search  
     * @param {log} log
     */
    function(record, search, log) {
        
        // Your code here
        
        return {
            beforeLoad: beforeLoad,
            afterSubmit: afterSubmit
        };
    }
);

Structure Breakdown

  • Array of strings ‚Äî Module paths (e.g., 'N/record', 'N/ui/serverWidget')
  • JSDoc comment ‚Äî Types for IDE autocomplete
  • Callback function ‚Äî Parameters match the array order
  • Return object ‚Äî Exports entry points that NetSuite can call

‚ö†Ô∏è Common Mistake

The order of modules in the array must match the order of parameters in the function. If you add 'N/email' to the array, you must add email as the next function parameter.

üö™ Entry Points

Entry points are functions that NetSuite calls at specific moments. Each script type has different entry points:

User Event Script Entry Points

Entry Point When It Runs Common Uses
beforeLoad Before record is displayed to user Add fields, buttons, hide fields, set defaults
beforeSubmit After user clicks Save, before database write Validation, calculations, data transformation
afterSubmit After record is saved to database Notifications, update related records, integrations

Client Script Entry Points

Entry Point When It Runs
pageInit When the page fully loads
validateField When user exits a field (before value is committed)
fieldChanged After a field value changes
postSourcing After field sourcing completes
saveRecord When user clicks Save (before submission)
lineInit When a sublist line is selected
validateLine Before a sublist line is committed

üìÑ The Context Object

Each entry point receives a context object with information about the current event:

function beforeLoad(context) {
    // context.type ‚Äî Event type: 'create', 'edit', 'view', 'copy', etc.
    // context.form ‚Äî The form object (can add fields, buttons)
    // context.newRecord ‚Äî The record being loaded
    // context.request ‚Äî HTTP request (for Suitelets)
    
    if (context.type === context.UserEventType.CREATE) {
        // Only run on create
    }
}

function afterSubmit(context) {
    // context.newRecord ‚Äî Record after changes
    // context.oldRecord ‚Äî Record before changes (null on create)
    // context.type ‚Äî Event type
    
    const recordId = context.newRecord.id;
}

‚úÖ Best Practice

Always check context.type before running logic. You rarely want a script to execute on every event type (create, edit, view, copy, delete).

üìö Finding Documentation

The NetSuite Help Center at docs.oracle.com is your primary reference. Here's how to find what you need:

For Script Types

Search for the script type (e.g., "SuiteScript 2.x User Event Script") to find:

  • Available entry points and when they trigger
  • Context object properties for each entry point
  • Supported event types
  • Code examples

For Modules

Search for the module name (e.g., "N/record module") to find:

  • All available methods and functions
  • Parameter requirements and return types
  • Governance units consumed
  • Usage examples

üìö Pro Tip: @see Tags

Add @see links in your script comments pointing to relevant documentation. This helps future developers (including yourself) quickly find reference material.

‚úçÔ∏è JSDoc for IDE Intelligence

JSDoc annotations enable autocomplete and type checking in your IDE:

/**
 * Validates and processes the expense report after save
 * @param {Object} context
 * @param {Record} context.newRecord - The saved expense report
 * @param {Record} context.oldRecord - Previous version (null on create)
 * @param {string} context.type - Event type (create, edit, delete)
 * @returns {void}
 */
function afterSubmit(context) {
    const { newRecord, type } = context;
    
    // Your IDE now knows newRecord has .id, .type, .getValue(), etc.
}

Common JSDoc tags:

Tag Purpose
@param {Type} name Documents a function parameter
@returns {Type} Documents return value
@throws {Error} Documents possible exceptions
@example Provides usage example

üéØ Key Takeaways

  • @NApiVersion 2.1 enables modern JavaScript features
  • @NScriptType tells NetSuite what kind of script this is
  • The define() block loads module dependencies in order
  • Entry points must be returned from the define callback
  • Context objects provide event type, record access, and form manipulation
  • Always check context.type before running logic
  • Use JSDoc annotations for IDE autocomplete and documentation
  • NetSuite Help Center is your primary documentation source

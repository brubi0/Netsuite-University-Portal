SuiteScript Basics

Understanding the fundamentals of SuiteScript development in NetSuite

üìÇ The File Cabinet

All SuiteScript files are stored in NetSuite's File Cabinet. To access it, navigate to:

Documents ‚Üí Files ‚Üí SuiteScripts

This is the primary location where SuiteScript files are stored when working in an account or transitioning between environments.

üí° SuiteApps vs SuiteBundles

Third-party applications come in two forms: SuiteBundles (bundled applications with objects and files) and SuiteApps (Oracle-approved marketplace apps). Both are stored in their respective folders in the File Cabinet.

Organizing Your Scripts

Best practice is to create subfolders within SuiteScripts to organize your projects. Each project should have its own folder containing all related script files.

The Last Modified date is helpful for tracking when scripts were updated‚Äîuseful when working in a team or reviewing your own Git history.

üìù Creating a Script Record

Uploading a file to the File Cabinet doesn't automatically make it a functioning script. You must create a Script Record to link the file to NetSuite's execution engine.

  1. Navigate to Customization ‚Üí Scripting ‚Üí Scripts
  2. Click the New button
  3. Select your script file from the File Cabinet
  4. Click Create Script Record
  5. Enter a meaningful Name and unique ID

‚ö†Ô∏è Naming Convention Tip

Add script type abbreviations to your names (e.g., "Create Field UE" for User Event, "Validation CS" for Client Script). This makes scripts easy to identify in long lists without opening each one.

Script ID Best Practice

NetSuite automatically prefixes your ID with custscript_. Use descriptive IDs that will remain unique across environments to avoid conflicts when moving scripts from sandbox to production.

üöÄ Script Deployments

A Script Record alone doesn't execute anything‚Äîyou need a Deployment to tell NetSuite when and where the script should run.

Field Purpose
Applies To Which record type triggers this script (e.g., Sales Order, Customer)
Deployed Checkbox‚Äîif unchecked, script won't run
Status Testing = runs only for you; Released = runs for defined audience
Log Level Controls which log types are recorded (Debug, Audit, Error, Emergency)
Audience Which roles, employees, or departments the script runs for
Execute As Role Permission level the script runs under (default: current user's role)

üìä Log Levels Explained

Log levels control what information is captured during script execution:

Level Use Case
Debug Development troubleshooting‚Äîvariable values, condition tracking
Audit Milestone tracking‚Äî"entered beforeLoad", "reduce stage complete"
Error Exception handling‚Äîcaught errors that need attention
Emergency Critical failures‚Äîsystem-level issues (rarely used)

‚úÖ Best Practice

In production, set Log Level to Error to reduce noise. Only use Debug when actively troubleshooting. Too many logs can make it hard to find real issues.

üë• Audience Configuration

The Audience tab on a deployment controls who the script runs for:

  • Roles ‚Äì Select specific roles or "All Roles"
  • Employees ‚Äì Target specific users
  • Departments ‚Äì Limit to department members
  • Subsidiaries ‚Äì Multi-subsidiary filtering

‚ö†Ô∏è Permission Consideration

Scripts are limited by the user's role permissions by default. If a script needs to create an Invoice but the user's role doesn't have Invoice access, the script will error. Use Execute As Role to run with elevated permissions when necessary.

üîç Execution Log & System Notes

After creating a script, two new tabs appear on the script record:

Execution Log

Shows real-time log entries from script execution. Filter by date, log level, or search for specific text. This is your primary debugging tool.

System Notes

Tracks all changes to the script record‚Äîwhen it was deployed, who modified it, and what changed. Everything in NetSuite is logged, so always ensure you have proper authorization before modifying production scripts.

üß™ Testing Your First Script

Once your script is deployed and released, test it by performing the action it's designed to trigger:

  1. For a User Event on Sales Orders: create or edit a Sales Order
  2. Check the record for expected changes (new fields, modified values)
  3. Review the Execution Log for any errors or debug output
  4. Verify the script ran for the expected user/role

In the example from the training, a script named "Create Field UE" added a new text field to Sales Orders. After deployment, navigating to Transactions ‚Üí Sales ‚Üí Enter Sales Orders revealed the new field with its default value‚Äîconfirming the script was working.

üéØ Key Takeaways

  • SuiteScript files live in the File Cabinet under /SuiteScripts/
  • A Script Record links your .js file to NetSuite's execution engine
  • Deployments determine when, where, and for whom scripts run
  • Use descriptive, unique IDs to avoid conflicts between environments
  • Log levels control what information is captured (Debug for development, Error for production)
  • The Execution Log is your primary debugging tool
  • Always test scripts with Testing status before releasing to all users

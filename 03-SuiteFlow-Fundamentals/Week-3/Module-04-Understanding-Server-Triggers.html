Server Triggers Overview

Server triggers represent events that occur on the NetSuite server rather than in the user's web browser. Unlike client triggers which only occur when creating or editing records, server triggers occur in all user event contexts‚Äîincluding when viewing existing records.

üí° Key Distinction

Client triggers = Browser events (create/edit only)

Server triggers = Server events (create/edit/view) ‚Äî Can also initiate workflows!

The Three Primary Server Triggers

Before Record Load

Occurs just before record data is displayed in the browser as part of the HTTP response. Allows pre-processing of the form prior to display.

Valid Contexts: Create, Edit, View

Common Uses: Default field values, change display characteristics, make fields mandatory/optional

Before Record Submit

Occurs just before the record is committed to the database. Since no data has been committed yet, you can cancel submission and return an error.

Valid Context: Save only

Common Uses: Form validations, calculations, field updates, business logic

After Record Submit

Occurs after the record has been successfully committed to the database. All saved data is available.

Valid Context: Save only

Common Uses: Create related CRM records (tasks, phone calls), send emails

‚úÖ No Database Rollback

NetSuite does NOT have a database rollback feature. A record is either fully committed or not committed at all. The server commits the record only when all Before Record Submit actions execute without errors.

State-Level Server Triggers: Entry & Exit

In addition to the three primary triggers, there are two special state-level triggers:

Entry Trigger

Occurs the first time a record enters a state. Actions execute only once when entering‚Äînot on subsequent visits or browser refreshes.

Exit Trigger

Occurs the first time a record exits a state. Actions execute only once when leaving‚Äînot if the record returns and exits again.

‚ö†Ô∏è Important: Entry/Exit vs. Browser Refresh

If users refresh their browser while a record is in a specific state, actions on the Entry trigger will NOT re-execute. Refreshing reloads the record in the current state‚Äîit doesn't force re-entry.

‚ö†Ô∏è Entry Trigger + Unsupported Actions

If a record enters a state on Before Record Load but has a Send Email action on Entry, the email will NOT send. Before Record Load is not a valid trigger for Send Email. The action will be skipped!

Workflow Trigger Event Sequence

Understanding the order of trigger events is critical for effective workflow design:

1
User requests record (new, view, or edit) ‚Üí Before Record Load fires
2
Page pushed to browser ‚Üí Before User Edit fires (client)
3
User interacts with fields ‚Üí Field-level client triggers fire
4
User saves ‚Üí Before User Submit (client) ‚Üí Before Record Submit (server)
5
Record persisted to database ‚Üí After Record Submit fires

Order of Action Execution in a State

  1. Entry trigger actions execute first (only once)
  2. Actions on the same trigger as entry execute next
  3. Client trigger actions as user interacts
  4. Before User Submit ‚Üí Before Record Submit on save
  5. After Record Submit after database commit
  6. Exit trigger actions before transitioning to next state

Client vs. Server Trigger Comparison

Feature Client Triggers Server Triggers
Where executed User's browser NetSuite server
Can initiate workflows No Yes
View context No Yes
Processing consistency May vary by browser Consistent
Best for Form interaction, validations Business logic, record operations

‚úÖ Best Practice

Use Before Record Load (server) instead of Before User Edit (client) for display characteristics‚Äîserver execution provides better processing consistency across browsers!

üåü Key Takeaways

  • Only server triggers can initiate workflow execution
  • Server triggers occur in all contexts (create, edit, view)‚Äîclient triggers only in create/edit
  • Use Before Record Load over Before User Edit for better consistency
  • Entry/Exit triggers execute only once per state visit
  • Browser refresh does NOT re-trigger Entry actions
  • No database rollback‚Äîrecord either commits fully or not at all
  • Use Before Record Submit for validations and field updates
  • Use After Record Submit for creating related records and sending emails
